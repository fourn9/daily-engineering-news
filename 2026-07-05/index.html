Cart Assistant: Agentic Grocery Shopping on Uber Eats

📐設計論🔥話題
出典: Uber Engineering Blog (2026-06-16)

Uber Eats のグロサリー用エージェント "Cart Assistant" のバックエンド設計解説。単一 LLM に丸投げするのではなく、multi-prompt state graph という設計 ── 計画 → 商品の関連性判定 → 数量選択 → 制約 (アレルギー・予算) 適用 → ガードレール → 評価、と 1 ステップ 1 プロンプト に責務を切り分け、それぞれが構造化 JSON を返し、決定的な API と検証層でつなぐ ── を採用。各ステップの trace が取れるため、"どの LLM 呼び出しで壊れたか" を eval で追える。

💡 補足: 現場でよくある失敗は "1 個の巨大プロンプトに全部詰めて、動くけど直せない" というもの。Uber の設計思想は "LLM 呼び出しをマイクロサービス化する" のに近く、Netflix の "小さな関数の合成" や DDD の bounded context 分割と同型の考え方を LLM ワークフローに持ち込んだ形。応用可能なエッセンスは "エージェントの各ステップを個別に eval して回帰を検出できる形にすると、変更コストが劇的に下がる" こと ── 自社の内製エージェントが 5 個以上の LLM 呼び出しを含むなら、この state graph 設計は輸入の価値が非常に高い。Anthropic 系の "sub-agent + skill" と組み合わせても機能する。

bliki: Agentic Programming

📐設計論🔥話題
出典: martinfowler.com / bliki 更新

Martin Fowler 本人が "Agentic Programming" ── 開発者がキーボードで書くのではなく、LLM エージェントに書かせて自分はレビュー・監督する開発スタイル ── を bliki の 1 エントリとして正面から定義した。従来の pair programming / TDD に代わる新しい work mode で、"人間が生成物を確認する頻度と粒度" が方法論の核だと整理している。同時期の "Interrogatory LLM" では、生成物を 問い詰める (interrogate) ことでモデル出力の信頼度を測る方法も提案。

Linux Foundation Announces Intent to Launch Agent Name Service (ANS)

📐設計論🆕新リリース⚠セキュリティ
出典: Linux Foundation / SiliconANGLE (発表 2026-06-23、Publickey 掲載 2026-07-02)

Linux Foundation が "AI エージェントを DNS で識別・検証する" 新オープン標準 Agent Name Service (ANS) を発表。既存の DNS 基盤の上に構築されるフェデレーテッドな identity レイヤで、"自社が持つドメインで自社エージェントを名乗る／署名する" 形にする。専用レジストリや中央集権プロバイダを介さない設計で、"どのエージェントが本物か / 誰に代理して喋っているか" をインターネット規模で検証可能にする。

From Signals to Structure: How Memory Architecture Drives Language Emergence in LLM Agents

🔬論文📐設計論
出典: arXiv 2607.00233 (2026-06-30)

LLM エージェントに "メモリ (会話履歴を超えて持ち続ける状態)" を持たせるとき、どんな構造で持たせるか が容量そのものより性能を左右する、という主張の実証研究。Lewis シグナリングゲーム (送信者・受信者が過去の相互作用だけを頼りに共通コードを合意していく) を使い、5 種類のメモリアーキテクチャを比較。永続的なプライベートノート (private notebook) を持つ agent は、チャンネル容量が余っても "高容量崩壊 (stateless agent で観測される、記憶が過剰でむしろ協調が壊れる現象)" を回避し、最も安定した協調を達成 (0.867 ± 0.023 @ capacity=25)。

💡 補足: 実務におけるエージェントメモリ設計 (Cursor の .cursorrules、Claude の Skills、Codex の AGENTS.md 等) は、経験則で "何をどこに置くか" が決まっているのが実情で、理論的な裏付け がほぼなかった領域。本論文の "容量より構造 / 特に個別 agent 専用の private notebook が効く" という所見は、"チーム共有の一枚岩ドキュメント" よりも "各 agent 個別の永続メモ" を優先する設計を後押しする ── これは Claude Code の subagent (それぞれが独立の履歴を持つ) 設計とも整合的で、"どの agent がどの memory を持つべきか" の設計の一次リファレンスになる可能性。

Claude Codeをチームで運用するためのCLAUDE.md設計とカスタムエージェント分担

📐設計論🔥話題
出典: Zenn (2026-06-03、継続的にランキング上位)

個人利用ではなく チーム開発で Claude Code を運用する ときに直面する現実問題を整理した実装ノート。CLAUDE.md (プロジェクト全体の共通指示) / custom agent (役割ごとの sub-agent) / MCP / Hooks の 4 つの仕組みを "どこに何を書くか" で役割分担させる方針を提案し、"よくある落とし穴 (規約の重複、context の肥大化、agent 同士の指示の食い違い) と回避策" を実運用の視点で列挙。

💡 補足: 上段の Martin Fowler "Agentic Programming" が方法論の抽象レベルの議論だとすると、この記事は "日本語チームで明日から適用する具体運用" レイヤの整理。特に "CLAUDE.md は理念、custom agent は職務分掌、MCP は権限、Hooks は自動化" というレイヤ分けは、社内で "AI 運用ガイドラインをどこに書くか" を議論するとき叩き台に使える。上段の openai/codex-plugin-cc と組み合わせると、"Claude Code を軸に置きつつ、必要な局面だけ Codex を呼ぶ" 実運用テンプレートが 1 枚に集約できる。

Claude Codeを手懐ける — CLAUDE.md・Hooks・Skills・MCPによるカスタマイズ戦略

📐設計論
出典: Zenn

Claude Code の拡張ポイント 4 つ (CLAUDE.md / Hooks / Skills / MCP) を、それぞれ "いつ何を書くか" の 意思決定木 の形で整理した実践記事。Skills は "手順 (playbook) をパッケージ化", Hooks は "決定的な自動処理を差し込む場所", MCP は "外部システムへのアクセス", CLAUDE.md は "全てのセッション共通の前提" と、住み分けを明確にする。

💡 補足: 前記事とほぼ同じテーマだが、こちらは 個人開発者が拡張の全体像を最短で掴む ためのマップとして機能する。上段の dotnet/skills と組み合わせると、"公式が用意した SKILL.md をどう自分の CLAUDE.md と協調させるか" の設計判断がやりやすい ── 具体的には "汎用ガイドは Skills として言語プラットフォームから輸入し、自社固有の意思決定だけ CLAUDE.md に残す" というレイヤ分けが実装可能になる。
