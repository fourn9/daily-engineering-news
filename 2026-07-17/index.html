Thoughtworks Technology Radar Vol.34 — "認知負債 (cognitive debt)" を中心テーマに据え、Zero Trust / DORA / testability への回帰を主張

📐設計論🔥話題ThoughtworksRadar
出典: thoughtworks.com/about-us/news (2026-04-15 公開 / 本ブリーフ初取り上げ)

Radar Vol.34 の中心は "エージェント時代の評価の難しさ / 原則は残しパターンは捨てる / 権限が肥大するエージェントの制御 / コーディングエージェントを "紐で繋ぐ"" の 4 テーマ。中核メッセージは "AI が生成コード量を増やすほど、人間と系との認知的距離が拡大し、cognitive debt (認知負債) が蓄積する" — 速度は "既に確立された技" (Zero Trust アーキテクチャ / DORA metrics / testability) への回帰で回収せよという方向転換。実験フェーズから成熟フェーズへの明示的な移行宣言。

💡読み手のアーキテクチャ関心にドンピシャ。汎用化のエッセンス — (1) "cognitive debt" というパターン言語化: 7/16 号の "context decay" と対になる新語で、technical debt が "コードとテストの負債" だったのに対し、cognitive debt は "人間の理解と系の実装の負債"。生成コード率が上がるほど負債が積み上がる = AI 時代特有の失敗モード。(2) "原則を残しパターンを捨てる" の含意: Microservices / Kafka / K8s / Kubernetes / GraphQL といった "パターン" は特定の時代のツールに紐付く。原則 (encapsulation / bounded context / observability / testability) は時代を超える。エージェント時代は "エージェントに何を書かせるか" を人間が理解できる抽象度で境界を切るのが本質、と Radar は主張。(3) 読み手の実務: 自社の設計レビューで "この機能に対する cognitive debt はいくらか" を新しい KPI として置く価値。具体には "生成コードの意図が人間に説明可能か / テストが振る舞いを言語化できているか / 権限境界が最小か" の 3 点を PR チェックに追加。7/16 の GitLab Report ("AI コードと人間コードを見分けられない 43%") への直接的な処方箋になる。

Pinterest Uses Content Fingerprints for URL Deduplication (InfoQ) — MIQPS が rendered content で "有意なクエリパラメータ" を機械的に判定

📐設計論Pinterestデータ基盤
出典: infoq.com/news (2026-06) / 本ブリーフ初取り上げ

Pinterest が MIQPS (Minimal Important Query Param Set) を公開。ルール表ではなく "rendered content fingerprint" で "このクエリパラメータはページ内容を変えるか" をオフラインで判定し、runtime で URL 正規化に使う二層アーキテクチャ。数百万ドメインを超える広域スケールで、重複処理コストを大幅削減。分類 = "コンテンツが変わる閾値以上 → 保持 / 未満 → ノイズ扱いで削除"。

💡読み手のアーキテクチャ関心にドンピシャ。汎用化のエッセンス — (1) "ルール表 → 学習ベース + オフライン計算" の設計置換: URL 正規化 (?utm_source= を削るか等) は 20 年間 "運用者が手でルール表を保守" が主流だったが、Pinterest はここに "実際にレンダリング結果が変わるか" という物理判定を持ち込んだ。同じ発想は "cookie 分類 / feature flag の効果測定 / A/B バリアント分類" に横展開可能。(2) オフライン学習 + runtime lookup の二層: 高コストな判定 (レンダリング比較) はオフラインで済ませ、実運用は "precomputed parameter map の keyed lookup" だけ = MLOps でよく議論される "expensive training + cheap inference" の抽象を、非 ML 系のパイプライン運用にも輸出できる好例。(3) 読み手の実務: 自社の SaaS で "重複 URL / 重複イベント / 重複メール" 系の除去に "ルール表 hell" があるなら、"content fingerprint で有意性判定 → 判定表を配信" 型に置き換えるのが有力。特に SEO / analytics / crawler / dedup が絡む処理で効果高。

Microsoft Launches Logic Apps Automation at Build 2026 (InfoQ) — agent-loop + Foundry agent + managed sandbox + Knowledge as a Service の 4 点セット

📐設計論Microsoftエージェント基盤

MemPoison: Uncovering Persistent Memory Threats and Structural Blind Spots in LLM Agents — arXiv 2607.14651

🔬論文セキュリティエージェント記憶
出典: arxiv.org/abs (2026-07-16)

7/16 の arXiv 2607.05029 (FARMA) に続き、LLM エージェントの永続記憶へのバックドア注入を扱う続報論文。MemPoison は 3 要素で構成: (1) semantic relational bridge = trigger と payload を意味的に紐付け同時に記憶抽出させる、(2) entity masquerading = trigger を "既知の固有名詞" に擬態させ書換抵抗性を得る、(3) joint embedding optimization = 埋め込み空間で trigger を密クラスタ化しつつ良性テキストから孤立させ stealth 化。既存 selective memory 防御を突破し攻撃成功率 最大 0.95。

💡読み手のための背景 — (1) 2026-07 の新攻撃面 "memory poisoning" が連続で公表: 7/16 の FARMA (推論トレースの偽造) → 7/17 の MemPoison (トリガーバックドア) と、1 週間で 2 本の重要論文が積み上がる pace。エージェント記憶 SaaS (Adaptive Recall / Mem0 系) を採用検討中の組織はセキュリティ質問リストを緊急更新すべきタイミング。(2) "selective memory" 防御の破綻: 既存対策は "怪しい記憶を選択的に無視" する方針だったが、trigger を entity masquerading で "見た目正常" に化けさせるとハイパープレーンでの区別が原理的に困難。防御は "埋め込み空間の anisotropy を陽に監視" / "書込監査ログの完全性" / "trigger 候補の adversarial training" 等の複合対策が必要。(3) 読み手の実務: 自社エージェントに長期記憶を持たせている / 検討中なら、記憶ストアへの書込ログを "誰が / いつ / どの文脈で" 監査可能に保持する優先度が急上昇。特に "顧客対応エージェントが記憶を蓄積するタイプの SaaS" では、顧客側の悪意ある会話が将来の判断を曲げる攻撃経路が現実的リスクに。
