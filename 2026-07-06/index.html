From Silos to Service Topology: Why Netflix Built a Real-Time Service Map

📐設計論🔥話題
出典: Netflix TechBlog (2026-06 末公開) / InfoQ 追報

Netflix の "数千のマイクロサービスの依存関係を、traffic の変化にほぼリアルタイムに追従する形で可視化する" 内製システム Service Topology の設計解説。eBPF によるネットワークフローログ (カーネルレベルの通信を計装なしで取得)、IPC メトリクス (RPC / gRPC 呼び出し統計)、分散トレース (Zipkin / Jaeger 系) の 3 系統を独立に集めた上で 1 つのグラフに統合し、"どのサービスが何に依存し、何から呼ばれているか" のトラバースクエリを 1 秒以下で返せる。動機は 4 年分のサポート依頼の解析で、"3AM の障害対応で on-call が最も聞いてくるのは これに依存しているのは誰か / これは何に依存しているか" だと分かったこと。

💡 補足: Netflix の話は往々にして "うちには関係ない" と受け取られがちだが、この記事の輸入価値は 3 系統ソースの独立設計 にある。従来型の service map は分散トレースだけに依存 (＝ トレースの計装漏れがそのまま盲点になる) か、Consul / K8s 由来の宣言的な依存だけを見る (＝ 実際に叩かれてない依存も出る) のどちらかで、結局 "実運用の依存関係" が正しく取れない。eBPF の "計装なし・実測ベース" と分散トレースの "リクエスト経路" と IPC メトリクスの "頻度と量" を 互いに独立な視点として突き合わせる ことで、片方が壊れても片方で検知できる冗長性が生まれる。テックリード視点では、"マイクロサービスが 20 個を超えた組織" ならこの 3 系統独立の思想は即応用可能 ── 完全な内製は不要でも、"eBPF ベースの実測依存 + トレースの経路" の 2 系統併用は今日から始められる。昨日の Uber Cart Assistant (LLM ワークフローをマイクロサービス化) の裏返しで、"膨大なマイクロサービスをどう見るか" の答えが Netflix 側にある。

How We Unified Configuration Distribution Across Systems at Uber (UCDP)

📐設計論
出典: Uber Engineering Blog

Uber 内では長年 Flipr (フィーチャーフラグ / config)、UBERCloud、Cassandra 系など複数の設定配信システムが並行して発達しており、"どれを使うべきか / どこに何を書くか" が事業ごとに散っていた。それを "設定を配信する仕組みは 1 個で十分" の原則で Unified Configuration Distribution Platform (UCDP) に統合。UCDP は 下層 (効率的・信頼性のある配信基盤) と 上層 (フィーチャーフラグ／dynamic config／実験など、ドメイン固有プロダクト) の 2 層に責務を分け、上層は差し替え可能、下層は 1 個の共通実装を全社で共有する。

💡 補足: "設定配信システムがサイロ化する" 問題は、Uber ほど巨大でなくても、社歴 10 年を超える組織で必ず起きる。ここで参考にすべきは 統合の切り分け方 で、"下層 (配信の効率・信頼性) と上層 (ドメイン UI / 用途) を分けて、下層だけ 1 本化する" 発想は輸入価値が高い ── 実務では "フィーチャーフラグを LaunchDarkly、実験は Optimizely、動的 config は自作 K8s ConfigMap …" と乱立するのが典型で、それらを API / UI 層は差し替え可能なまま、配信レイヤだけ 1 つに統一する と、SLO 管理・監査・キャッシュ戦略が単一箇所に集中できる。逆に "全部 1 個のツールに寄せる" と失敗しがちで、そちらの反面教師でもある。

Next.js 16.3 — Turbopack: 90% メモリ削減・永続ビルドキャッシュ・Rust 版 React Compiler

🆕新リリース🔥話題
出典: nextjs.org/blog (2026-06-29 リリース、7 月に fixup 追報)

Next.js 16.3 が Turbopack 中心の 3 本柱で刷新。(1) メモリ eviction ── 長時間 dev サーバを立てているとメモリが積み上がる問題を修正、実測で 最大 90% 削減 (設定不要でデフォルト有効)。(2) persistent build cache ── turbopackFileSystemCacheForBuild フラグで .next をビルド間で使い回せる。CI で前回の .next をキャッシュ ⇒ 復元するだけで大幅短縮。(3) Rust 版 React Compiler ── これまで Babel 変換で動いていた React Compiler が Rust ネイティブに移植され Turbopack パイプラインに統合、大規模アプリ (Vercel 内部の v0) で 20〜50% のコンパイル時間短縮を確認。

💡 補足: "React Compiler" とは React 19 で導入された "手書きの useMemo / useCallback の大半を自動化するコンパイラ最適化" のこと (＝ 開発者が memo をつけ忘れて再レンダリングが起きる問題を、AST 解析で先回りする)。今回のポイントは "その最適化が Rust 版で速くなった" 以上に、Babel を通らずに Turbopack から直で叩けるようになった こと。従来は Turbopack (Rust) が Babel (JS) の完了を待つ形で、Rust 側の並列度を活かせなかった。昨日号の Astro 7.0 も "コンパイラを Rust ネイティブに書き換え、fetch handler 標準に合流" と同じ潮流で、"JS/TS ツールチェインの内部言語が Rust に統一される" 大局は Next.js 側でも完成に近づいた。テックリード視点では、CI 時間の削減効果が大きく、Turbopack への移行判断のハードルが 1 段下がった。

AWS Lambda MicroVMs — 状態を持てる Firecracker サンドボックス (最長 8h)

🆕新リリース🔥話題📐設計論
出典: AWS News Blog (2026-06-22 launch、7 月継続話題)

AWS Lambda に MicroVMs という新プリミティブが追加。Lambda の内部で使われている Firecracker (Amazon 製の軽量 VM 技術。月あたり 15 兆回超の Lambda 実行を支える) を直接プリミティブとして開放し、"VM レベル隔離 (共有カーネルなし)、ほぼ即時 (数百 ms) の起動と resume、最長 8 時間の実行、ARM64 / 最大 16 vCPU / 32GB mem / 32GB disk" のサンドボックスを Lambda 経由で作れる。用途としてターゲティングされているのは AI コーディングアシスタント / インタラクティブなコード実行環境 / データ分析 / 脆弱性スキャナ / ユーザースクリプトを走らせるゲームサーバ。

💡 補足: これは "エージェント時代のインフラプリミティブ" として非常に大きな意味を持つ発表。従来 "AI エージェントに任意のコードを実行させる" には、コンテナ (Docker / K8s / ECS) を都度立ち上げるか、E2B / Modal / Daytona 等のサードパーティ製サンドボックスを使うしかなく、"起動 5〜30 秒 + セッション状態が消える + Lambda の 15 分制限に引っかかる" のどれかで詰まっていた。MicroVMs は "状態が残せる (最長 8h)、Lambda 課金モデルで安い、EC2 のような管理不要" の 3 拍子で この構造課題を丸ごと解く 提案。テックリード観点では、"社内エージェントに任意コード実行させる基盤" を Modal / E2B に外注していた組織は、AWS 内で完結させる選択肢が現実的に。加えて、"ユーザーがブラウザ上で任意コードを試せる SaaS" (Replit 系) を作る組織にとっては、コスト構造が大きく変わる。同日リリースの Bedrock 側 Fully Managed Knowledge Bases と組み合わせると、"RAG + 実行環境" が Lambda 1 個の設定で組める。

Notion Workers — Notion 側で TS/JS を動かすホスト型ランタイム (5 月発表・7 月更新)

📐設計論🆕新リリース
出典: Notion Developer Platform Release Notes / announcement blog (7 月に PAT 有効期限機能追加)

Notion が発表した Developer Platform 3.5 の目玉である Workers。TypeScript/Node.js コードを Notion 側のインフラでホストできる serverless runtime で、webhook を受け → 任意ロジック → Notion 側の DB / ページ操作 or 外部 API 呼び出し、が "自前のサーバを一切立てずに" 実装できる。同時に External Agents API が Claude Code / Cursor / Codex / Decagon を "workspace の一級参加者" として扱う仕組みを提供、Database Sync が任意の REST/GraphQL API を Notion DB にライブ同期する。ベータ期間は無料、8/11 以降 Notion クレジットで課金。

EU Council forces Chat Control via fast-track

⚠セキュリティ🔥話題
出典: heise (2026-07-05) / Hacker News 前ページ掲載

EU 理事会が児童性的搾取物 (CSAM) 対策を口実とする Chat Control (CSAR) 規則を再度早期採決トラックに乗せた、と heise 記事が報告。今回の草案では暗号化されたメッセージング (Signal / WhatsApp / iMessage / メールも含む) に対し、クライアント側でのスキャン (client-side scanning) を義務化する条項が復活。Signal / EDRi / EFF などがこれまで通り "実質的にエンドツーエンド暗号化の破壊" と反対。今回は fast-track を使い、通常より短い審議期間で合意を狙う点が新しい。

💡 補足: Client-side scanning (CSS) とは "暗号化される前のメッセージを、送信者の端末上でハッシュ照合・ML 検出する" 方式。エンドツーエンド暗号 (E2EE) は "サーバ側は暗号文しか見えない" 保証だが、CSS は端末側で見るため理屈上 E2EE を破らずに検閲できる、と主張される。反対派の論点は 2 つ: (1) 一度端末に検閲コードを置くと、対象を CSAM 以外にも拡張する法改正だけで済む (機能自体は既にある)、(2) ハッシュ DB は非公開になりがちで、"何が検閲対象か" が民主的にコントロールできない。テックリード視点では、自社が EU 市場向けメッセージング / チャット機能を持つ場合、法制化されると "実装する / EU 市場から撤退する / E2EE を弱める" の三択を迫られる可能性が高い。加えて、Signal は過去に "EU が CSS を義務化するなら EU から撤退する" と表明済みで、"どの SaaS が EU で使えなくなるか" は今後 6〜12 ヶ月で判断が要る領域。

When Agents Do Not Stop: Uncovering Infinite Agentic Loops in LLM Agents

🔬論文📐設計論⚠セキュリティ
出典: arXiv 2607.01641 (2026-07-02、Xinyi Hou ら)

"エージェントが止まらない" 障害クラス Infinite Agentic Loops (IALs) の系統的な分類論文。従来のプログラムの無限ループとは異なり、IAL は "モデル呼び出し・ツール呼び出し・workflow 遷移・agent 間の handoff" が フィードバック経路が有界化されていない ために停まらない現象で、"1 リクエストがコスト枯渇 / model の DoS / context 膨張 / 副作用の反復" に化ける。論文はエージェントロジック × フレームワーク意味論 × ランタイム観測 × 終了機構 の 4 軸で IAL を分類し、実 fw (LangChain / AutoGen 等) で発生パターンを検出。

💡 補足: 実務でエージェントを本番投入している組織なら、"止まらないエージェント" は既に一度は踏んでいる問題 (ツール呼び出しが失敗 → 再試行 → 別ツール → 元に戻る、や、複数エージェントの handoff がループする、等)。この論文の価値は "IAL は無限ループの延長ではなく フィードバック経路の設計失敗 だ" と概念整理し、"何を有界化すべきか" の設計チェックリストを与える点。テックリード視点で、内製エージェントの本番投入前に (a) 各サブエージェントの最大 iteration、(b) ツール呼び出しの budget、(c) handoff の depth、(d) context growth の閾値、の 4 つは必ず定義する運用にすべき ── 昨日話題になった Uber Cart Assistant の state graph 設計は、この "有界化を graph 定義で強制する" 側からの回答とも読める。

AgentFlow: Building Agent Dependency Graphs for Static Analysis of Agent Programs

🔬論文📐設計論
出典: arXiv 2607.01640 (2026-07-02)

エージェントプログラム向けの 初の静的解析フレームワークを提案。Agent Dependency Graph (ADG) という framework 非依存の表現を導入し、agents / prompts / models / capabilities / memory / control policy を typed node として、component-dependency / control-flow / data-flow を typed edge として捉える。ADG から Agent BOM (Bill of Materials — エージェント構成の部品表) を自動生成でき、"prompt から tool への意図しないアクセス経路 (prompt-to-tool risk)" を静的に検出。5 つの代表的な fw (LangChain / AutoGen / CrewAI / LangGraph 等) 上で 5,399 個の実世界エージェントプログラムを評価し、238 件の実際の prompt-to-tool リスクを検出。既存の Agentic Radar / Agent-Wiz より詳細で、Drako / Trusera / Cisco 系の AI-BOM ツールより情報量が多い、と主張。

💡 補足: "Agent BOM" は SBOM (ソフトウェア部品表) の agent 版で、SBOM が npm / OSS の依存関係を静的に列挙するのに対し、Agent BOM は "このエージェントが呼ぶ tool / 参照する prompt / アクセス可能な memory" を列挙する。同日の "IAL 論文" が ランタイムの停止性 の話だったのに対し、こちらは デプロイ前の静的検査 の話 ── 対をなす。テックリード観点では、エージェントを 5 個以上運用する組織は、CI パイプラインに "ADG 抽出 → 意図しない tool 到達性の検査" を組み込む価値が急速に上がる。従来 "prompt injection で任意の tool を叩ける" 種の脆弱性はレビュー時に見落とされがちだったが、それを コンパイル時に検出する 発想は、Trivy / Snyk が SBOM を検査するのと同じ抽象化。"エージェントを扱う組織の DevSecOps はこの形になっていく" というシグナルとして注目。
