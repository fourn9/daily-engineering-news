📐 アーキテクチャ / 設計 — Netflix の in-house LLM serving と Fowler "DSL による LLM の飼い慣らし"

In-House LLM Serving at Netflix (Netflix TechBlog / AI Platform Model Runtime team) — MSS (Model Scoring Service) + NVIDIA Triton + vLLM で XGBoost/TF/PyTorch/LLM を単一 IF、OpenAI 互換 API を gRPC と並列に露出

📐設計論🔥話題NetflixLLM ServingvLLM
出典: netflixtechblog.com (2026-07-18 公開 / 本ブリーフ初取り上げ)

Netflix が LLM を hosted API 経由ではなく自前 stack で運用する意思決定と、その具体的な構成を公開。Model Scoring Service (MSS) を "XGBoost/TensorFlow/PyTorch/LLM を統一 IF で serving" するバックエンドとして位置付け、下地は NVIDIA Triton Inference Server、LLM エンジンは vLLM を "paved-path" として採用 (2025 夏時点で OSS エンジンが専用スタックとの性能差をほぼ埋めたと判断)。OpenAI 互換 API を gRPC と並列に第 2 の frontend として露出し、LLM エコシステム全体のクライアント互換性を担保。上位 JVM 側は routing / A/B / 候補生成 / feature fetch / 推論 / 後処理 / ログを一貫して扱う。

💡読み手のアーキテクチャ関心にドンピシャの一次資料。汎用化のエッセンス — (1) "LLM を新規サブシステムにしない" 設計判断: 多くの企業は "LLM serving" を推薦系 ML と別 stack で立ち上げるが、Netflix は 「LLM は "4 つ目のフレームワーク" として既存 MSS の上に載る」という判断を採った。routing / A/B / 監視 / feature fetching を全モデル種で共通化できるので運用ヘッドカウントが線形に増えない。7/17 Pinterest AGN のように "エッジ推論を汎用平面に統合" した動きと同種で、"AI/ML 基盤は "1 つのプレーン" に統合するのが 2026 後半のパターン"。(2) vLLM を "paved-path" として選ぶ意味: vLLM = UC Berkeley 発の OSS 高速 LLM 推論エンジンで、continuous batching・PagedAttention で 2024 以降デファクト化した。専用スタック (TGI/TensorRT-LLM) が優位だった時代は終わったので "運用適合性" (custom model の compile pipeline 不要) で選ぶフェーズに入ったという宣言でもある。7/13 の "OSS ranking" の議論と同じ結論。(3) OpenAI 互換 API を第 2 frontend で並列露出する運用ノウハウ: 内部の "純正 API" (gRPC) は最適化して残しつつ、外部の SDK / MCP tool / エージェント / eval フレームワークが期待する de-facto 標準 = OpenAI 互換を必ず並列で用意する = 内部性能を落とさず外部互換性を担保する二重化。読み手が自社で "LLM 基盤" を作るなら、"1 日目から OpenAI 互換 endpoint を用意する" のは (品質論ではなく) エコシステム摩擦を減らすための現実解。(4) Triton を選ぶ理由: モデル読込・batching・GPU スケジューリングを OSS で持っていける = ベンダー依存を最小化しながら "1 GPU に複数モデル同居" を安定運用できる。7/19 Cloudflare の "OSS への還流" と対で、ハイパースケール企業が "自社最適 stack を捨てて OSS を運用しながら育てる" 動きの最新事例。(5) 読み手の実務: 自社が推薦系 ML + LLM の両方を運用しているなら、この "1 つの serving プレーンに統合する" 発想を まず 1 週間で "MSS 相当の統一 IF レイヤ" を型定義から書き出す だけで議論の起点になる。

DSLs Enable Reliable Use of LLMs (Martin Fowler / Kenny Baas-Schwegler) — Tickloom を題材に "DSL は LLM を安全に飼い慣らす harness" と定式化、few-shot で構文を確実に生成させる回路

📐設計論FowlerDSLLLMharness
出典: martinfowler.com/articles (2026-07 公開 / 本ブリーフ初取り上げ)

7/14 の Fowler フラグメント "DSL / 抽象が LLM の harness になる" を 本編論文として展開。分散システム挙動を記述する DSL Tickloom を素材に、"LLM を DSL の設計パートナー として使いつつ、DSL 自体を LLM が確実に正しい構文を吐くための "in-context example が効きやすい制約された言語" として設計する" 双方向の使い方を提示。汎用 Java / Python では variation が大きすぎて few-shot が効きにくいのに対し、DSL は "汚染源の少ない言語" として LLM の hallucination を構造的に抑える。

💡読み手のための背景 — (1) 7/13 → 7/14 → 7/19 の Fowler 3 部作の中核: 7/13 "harness engineering" (足場) → 7/14 "DSL は harness の 1 種" (フラグメント) → 7/19 arXiv 2607.13683 "harness の自己進化" (学術化)、その中間の "harness を具体的にどう設計するか" の実装論が本記事。(2) "variation を減らす" が LLM 信頼性の本質: 汎用言語は同じ処理を 20 通り書けるが、DSL は 1〜2 通りに絞れる = LLM の生成分布が事前に狭いので、few-shot 例と実生成のカバー範囲が一致しやすい。7/16 DoorDash eval-as-CI や 7/19 の Cloudflare 事案とも共通のパターン = "AI 出力の品質は "モデルの賢さ" ではなく "出力空間の設計" が支配的"。(3) 読み手の実務: 自社が LLM で "契約書のパース / ワークフロー宣言 / 監視ルール生成" を扱うなら、"自然言語 → 汎用コード" ではなく "自然言語 → 小さな DSL → 実行" の二段構えにすると桁違いに安定する。DSL のホスト言語 (YAML / JSON / TypeScript 型 / Lean) 選定は "LLM が最も型ミスをしにくい" 基準で決めるのが 2026 後半の実務パターン。(4) Tickloom の意味: Fowler が "自分で作ってみせた分散システム挙動記述 DSL" = 「DSL 設計自体を LLM とペアで進める」体験談を "生きた素材" として同時公開する手口。読み手が自社ドメインで真似しやすい "設計プロセスの写経元" として使える。

AWS Lambda Managed Instances (AWS re:Invent 2025 発表 → 2026 Q1〜Q3 段階拡張) — 関数を "自社 AWS アカウント上の EC2" で動かす、AWS が OS/patch/LB/autoscale/runtime だけ管理

📐設計論AWSLambdaServerless
出典: docs.aws.amazon.com / re:Invent 2025 発表 / 2026 展開継続中 (本ブリーフ初取り上げ)

従来 Lambda が "AWS 側 fleet に相乗り" だったのに対し、Managed Instances は Lambda 関数を ユーザ AWS アカウント内の EC2 インスタンス にデプロイする方式。ユーザは instance type / min-max fleet size / dedicated tenancy を指定し、OS パッチ / セキュリティ更新 / fleet 内 LB / autoscale / Lambda runtime は AWS が引き続き管理。256 KB payload 上限撤廃・durable functions (multi-step / AI workflow)・SQS provisioned mode と合わせた "2026 serverless stack" 再定義の 1 ピース。

💡読み手のための背景 — (1) "Serverless の再定義" の意味: 従来 Lambda = "リクエスト単位課金・秒単位起動・256 KB payload" が制約だったが、AI/LLM ワークロード (大きな payload・長時間・GPU) や regulated tenant (専有 EC2 が必要) が Lambda の外に流出していた。Managed Instances は "Lambda の DX を保ったまま Isolation / 大サイズ / 長時間実行が必要な workload を吸収し返す" 反撃。(2) 汎用化のエッセンス: 2026 の serverless は "純粋 FaaS ↔ 純粋 IaaS" の二極から "hybrid tier" に転換。GCP Cloud Run Sandbox / Azure Container Apps も同じ方向。読み手が新規 workload を FaaS で設計する時は "durable + managed instance + provisioned SQS" の 3 点セットで書ける前提で AWS リファレンスアーキテクチャを再学習しておく価値。(3) 読み手の実務: 特に agentic workflow (LLM tool call が長時間になる) を今から書くなら、Step Functions + Durable Functions + Managed Instance Lambda + SQS provisioned の組み合わせが 2026 後半のリファレンス。従来 "Lambda に載らないので ECS に逃す" と決めていた設計を改めて Lambda 側で試す価値。(4) bounded context 原則の serverless への輸出: 記事は "cart 関数群 = 1 SAM template / 1 リポ" のように マイクロサービスの bounded context 原則を関数のフォルダ / repo 分割にそのまま適用する という現代の serverless 組成手順を明文化。関数を "全社共通の 1 リポ" に押し込むアンチパターンから離脱する根拠として使える。

SpaceXAI open-sources Grok Build after data-retention scandal (The Register / 7/15 OSS 化) — CLI が Git repo を丸ごと SpaceXAI GCS bucket に無許可 upload (5.1GB / 26,000× 過剰)、72 時間後に Apache 2.0 で公開・upload コードはまだ同梱

⚠セキュリティ🔥話題GrokSpaceXAIharness
出典: theregister.com / axios.com / github.com/xai-org/grok-build (2026-07-12〜2026-07-15)

7/12、研究者 Cereblab が Grok Build CLI v0.2.93 が "no files を指示していても" 全 Git repo (unredacted secrets + full commit history) を SpaceXAI 所有 GCS bucket に silently upload していることを wire-level トラフィック解析で公開。192KB のタスクに対して 5.1GB (約 26,000 倍) upload。SpaceXAI は 7/12 に retention をデフォルト無効化、7/15 に Grok Build 本体を Apache 2.0 で github.com/xai-org/grok-build に OSS 化 (Rust harness)。だが repo は squashed monorepo mirror・external contribution は closed・scandal の元凶だった upload machinery はまだコードに残っている。

💡読み手のための背景 — (1) "CLI が repo 丸ごと送る" の技術的意味: Coding agent CLI (Claude Code / Cursor / Codex / Kimi CLI / Grok Build) は コンテキスト提供のために prompt に含むファイルを "工夫して選ぶ" 設計が本来なので、"192KB のタスクに 5.1GB 送る" は明らかな設計ミスまたは意図的な telemetry。7/13 arXiv 2607.04528 "harness が信念軌跡を発散させる" と対で、harness 側が data exfiltration の主要経路になり得ることの実例。(2) "72 時間で OSS 化" の意義と限界: SpaceXAI が Apache 2.0 で公開したのは trust 再建の速度としては近年最速級だが、(a) squashed history では過去の telemetry コードは追えず、(b) upload machinery が残ったままなので「見て納得はできるが根本修復ではない」。7/18 の GitHub Code Quality GA と同じ "gate 型 governance" の逆パターン = 透明性は上げたが実際の behavior は変わっていない。(3) 読み手の実務: (a) 開発者マシンで動く coding agent CLI (全ブランド) は egress 監査を実施 = "task に無関係な GB 単位の HTTPS out がないか" tcpdump / mitmproxy で確認。(b) 「AI CLI に "全 repo を送るな" と指示しても信じない」を前提とし、ネットワーク側で egress allowlistを掛けるのが 2026 後半の共通線 = 7/18 Grab Palana Envoy egress proxy と同じ解。(c) 社内での Grok Build 採用議論は upload machinery が repo から削除されるまでは "評価段階" に留めるのが妥当。(4) 汎用化のエッセンス: Coding agent の "信頼性" は "OSS かどうか" ではなく "実際の egress が制御できるか" で測る時代 = OSS 化はスタートラインで、ゴールではない。

⚠セキュリティ🔥話題GitHub ActionsCI/CD

💡読み手のための背景 — (1) Bun 移行事例の意味: Zig の学習曲線と rust の maintainer プール差が Bun 側で顕在化。Claude を "翻訳器" として使い 1 万行超の Zig → Rust 移植を段階的に実施する事例は、「大規模移植を LLM で実施する 2026 後半のリファレンス」として学ぶ価値。7/15 の Cloudflare hyper 移行話とも系譜が同じ。(2) Multigres の位置: Postgres を "sharded / horizontal" にする OSS が Cluster / Citus / CockroachDB / YugabyteDB に続いて Multigres で 5 番目。Vitess (MySQL) の Postgres 版という定式は分散 DB 市場に "選択肢の重複" を生む = 選定基準は "コミュニティ活性度 + 商用サポート" で見るフェーズ。(3) PostgreSQL 19 beta の含意: I/O worker autoscale は "書き込み負荷の増減に応じて worker 数を自動調整" する = Aurora / GCP AlloyDB が持っていた "自動チューニング" を OSS 本体が持つ。読み手が RDS / Aurora / CloudSQL を検討する場合、"OSS 単独でもかなり戦える" ラインが上がった。(4) Cloud Run Sandbox の意味: 「AI が生成したコードを安全に実行する」専用サンドボックスは、7/19 の "SaaS の MCP 化" と対で エージェントが生成した実行コードを SaaS 側で受け入れる基盤として重要。7/20 の Grok Build 事案 (上記) を裏返せば "サンドボックス側 = 実行者側の防御" が課題。
