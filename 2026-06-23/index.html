Cursor "Origin" / GitLab "Project Switch" / Zed "DeltaDB" — AIエージェント向けGit再構築レース

🔥話題📐設計論The New Stack · 2026-06

3社が同時期に「GitHub は AI エージェントが量産するコミット/PR量に耐えられない」という前提で次世代 Git 互換バックエンドを発表。GitLab の Project Switch は分散アーキテクチャでエージェントタスクを最大 50倍、書き込みを 17倍 高速化、トークン使用量も半減を謳います。GitHub にロックインされている組織にとっては数年スパンの選定見直しトリガになる可能性。

Claude Code Skills / MCP / Subagents の使い分け（2026版整理）

📐設計論Totalum Blog 等 · 2026-06

2026年現在の実務指針:「外部システムへのアクセスは MCP、手順知識は Skill、大きなコンテキスト分離は Subagent」。MCP は GitHub/Postgres/Linear/Sentry のように外部システムごとに1個ピン留めし、その上で薄い Skill を書いて手順をオーケストレーションするのが定着しつつあるパターンです。

Skill は使うまで約100トークンしか食わないのがポイント。50個入れていても呼ばれない限りコストはほぼゼロなので、"とりあえず詰める" が許容されます。MCP（capability）/Skill（knowledge）/Subagent（context window 分離）という3軸で整理しておくと、社内エージェントの拡張時に迷いません。

Martin Fowler — bliki: Architecture Decision Record（再注目）

📐設計論martinfowler.com

「ADR は数ページの短い文書で、決定・文脈・重大な影響だけを書く。決定が変わったら ADR は 修正せず 新しい ADR にリンクさせる」というシンプルなルール。AI エージェントによりコード変更速度が増す中、"なぜそうなっているか" を後で追える ADR の運用価値が再評価されており、Thoughtworks Radar 等でも継続的に Adopt 推奨されています。

ADR の本質は「議事録」ではなく「ロックイン宣言」。RDB を Postgres に決めた・認証を OIDC で統一した、のような「後から覆すと高い決定」だけを記録するのがコツで、些末な選択まで書き始めると形骸化する。AI agent が PR を量産する時代では、ADR が "agent に守らせる制約集" としても機能し始めています。

🗄️ データベース / データ基盤

DuckDB 1.5.4 リリース、pg_duckdb は Azure HorizonDB プレビューに同梱

🆕新リリースDuckDB · 2026-06-17

DuckDB 1.5.4 はバグ修正と機能改善が中心。注目は pg_duckdb 拡張で、PostgreSQL の OLTP 上に DuckDB の OLAP 速度を載せられる組み合わせが Azure マネージド PostgreSQL（HorizonDB）の標準拡張群（約75本）に入りました。OLAP 専用 DWH を持たない中小規模では選択肢が大きく変わります。

これまで PostgreSQL 上で分析クエリを速くする選択肢は Citus (シャーディング) か外部DWH(Snowflake/BigQuery)転送が中心でしたが、pg_duckdb は 同じ Postgres 接続から DuckDB エンジンに直接クエリを流せるのが新しい。データ移動コストとデータ重複を持たずに済むため、analytics の "とりあえず Postgres でいい" 領域が広がります。

🔬 注目論文 / 研究

Multi-Agent Transactive Memory (Carnegie Mellon × UC Berkeley)

🔬論文arXiv:2606.19911 · 2026-06-18

「能力の異なる LLM agent を分散して動かす際に、agent 間の知識共有をどう組み立てるか」を扱った論文。検索エンジンが人間の生成物を整理してきたのと同じように、agent が生成した artefact を agent 集団間で再利用するための retrieval 基盤を提案。社内に多数の agent を抱える運用フェーズに入る際の参照価値があります。

"Transactive Memory" は社会心理学の概念で、組織内で「誰が何を知っているかを知っている」状態のこと。これを agent 間で実装するという問題設定は、社内で 10〜100 個の agent を並走させる組織にとって直接的な設計テーマで、MCP の "tool catalog" を超える meta-layer 議論の出発点になります。
