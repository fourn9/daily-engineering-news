🛠 AIエージェント / 開発インフラ

Run isolated sandboxes with full lifecycle control: AWS Lambda introduces MicroVMs

🆕新リリース📐設計論
aws.amazon.com｜2026-06-23

AWSがLambdaに新プリミティブ「MicroVMs」を追加。Firecracker上に最大8時間のステートフル実行環境を提供し、コンテナより強いVMレベル隔離をサブ秒起動で実現。コーディング環境・データ分析・脆弱性スキャナなど"ユーザー/AI生成コードを動かす場"を、自前のVM管理なしに作れる。

従来のLambda関数は「15分上限・ステートレス」で、AIエージェントの長時間ループや会話セッションには窮屈だった。MicroVMsは"Firecracker（軽量VM）の Lambda 側へのリパッケージ"で、E2BやFly Machinesといった先行サービスが切り拓いた"エージェント用サンドボックスSaaS"のレイヤを AWS が公式機能として吸収する形。AIエージェント基盤の自作判断に影響大。

Model Once, Represent Everywhere: UDA (Unified Data Architecture) at Netflix

📐設計論
netflixtechblog.com｜2026年6月

Netflixが社内データ基盤のメタアーキテクチャ「UDA」を公開。RDF/SHACLベースのknowledge graphをハブに据え、ドメインモデルからGraphQL/Avro/Iceberg/Javaスキーマを自動派生。"actorsやmovies"といったビジネス用語でデータを探せるセルフサービス検索ツール Sphere をこの上に構築している。

大規模組織で必ず起きる「同じ概念に部署ごとの別スキーマが乱立する」問題への構造的回答。Data Mesh（分散所有モデル）やData Catalog（後付け検索）と違い、UDAは"スキーマを上位ドメインモデルから生成する"発想で、データプロデューサ側を矯正する。中小規模でも"単一ソースからAPI/DB/型を生成する"発想は適用でき、設計判断の参考価値が高い。

Apache Hudi™ at Uber: Engineering for Trillion-Record-Scale Data Lake Operations

📐設計論
uber.com｜2026年6月

Uberがtrillion-record規模でApache Hudiを運用してきた工夫を解説。データレイクの一部をクラウドへ無停止移行し、PB級レプリケーションを分散Application Master化＋並列namenode呼び出しで90%高速化、5倍のレプリ容量を実現したと報告。

HudiはUberが社内基盤として開発・OSS化した「データレイクにACID/UPSERTを足す」フォーマットで、Apache Iceberg／Delta Lakeと並ぶlakehouseの3大選択肢。仕様の話ではなく"原開発元がhyperscaleでぶつかった運用課題"を共有している点が読みどころで、Iceberg採用検討中でも"テーブル/メタデータ無停止移行"の論点として参考になる。

Less Context, Better Agents: Efficient Context Engineering for Long-Horizon Tool-Using LLM Agents

🔬論文📐設計論
arXiv:2606.10209｜2026-06-08

企業システムの饒舌なツール応答がエージェントの context overflow / stale-state エラー / 推論コスト増を招く問題を、MCP（Model Context Protocol）ツールとGPT-5系で実測。複数のcontext管理戦略を比較し、"渡す情報を絞る方が"精度・コストの両方で勝るケースが多いことをホテル経費ベンチで定量化。

「長コンテキストモデルなのだから全部突っ込めばよい」という素朴な発想を否定する流れに、MCPという文脈で具体的なベンチを与えた1本。MCPは外部ツール群を統一規格でエージェントに繋ぐ仕組みで、Claude/GPT/Gemini等が広く対応中。ツール応答の要約・トリミング・選択をどこに置くかは、エージェント設計の中心論点になりつつある。

モジュラモノリスで急成長SaaSをスケール - マイクロサービスは"実装の詳細"

📐設計論
zenn.dev / NeoAI

DDDによる強い境界を持つモジュラモノリスを採用すれば、"マイクロサービスかモノリスか"の選択は実装詳細に降格でき、将来の切り出しを低コストで可能にできる、という運用知見の共有。急成長フェーズの実プロダクトで採った設計判断とトレードオフが言語化されている。

"最初からマイクロサービス"で失敗してきた業界経験への一つの解。Sam Newmanの「modular monolith → 必要部分だけ分離」進化型アーキの実装事例で、初期フェーズで境界（モジュール内/間の依存ルール、DBスキーマ、トランザクション境界）を厳格に運用できれば、後の分割コストは劇的に下がる。中規模SaaS立ち上げ期の設計判断テンプレとして直接参考になる。
