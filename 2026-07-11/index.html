🧰 AI が言語・DB を書き換える (Bun の次の証拠)

malisper/pgrust — Postgres 18.3 の回帰テスト 46,000 件を 100% パスする Rust 版 Postgres (Codex 8 並列 / 月 $1,600 / 個人開発)

🔥話題🆕新リリースRust📐設計論HN #1
出典: GitHub malisper/pgrust / malisper.me 解説記事 / Hacker News (7/9 announced, 7/10 #1 で 740 pt / 620 コメント)

Malcolm Matis が個人開発で "Postgres を Rust で 1 から書き直し" を成立させ、7/9 に v0.1 を公開。46,000 件の Postgres 18.3 回帰テストを 100% パス、既存の Postgres 18.3 データディレクトリからそのままブート可能というディスク互換を実現。公開版は 15 万行超、内部未公開版は 45 万行超・"thread-per-connection モデル" (process-per-connection ではない)、トランザクションで +50%、分析ワークロードで ~300 倍高速の主張。実装は Codex 8 アカウント ($200/mo × 8 = $1,600/mo)、"Conductor" と呼ぶ自作ツールで 10〜20 セッションを並列制御。README では "まだプロダクション用途ではない・性能最適化前" と明記。

💡昨日の Bun (Anthropic + Fable 5 / 64 並列 / $165K で 11 日 / 535K 行) と対をなす事例で、"個人開発でも AI に大規模書き換えを任せられる" 下限が今週で見えた。汎用化可能なエッセンス — (1) 回帰テストを "AI の合格判定" として使い切る Oracle 型設計: Matis は自ら仕様を書かず "既存の Postgres 回帰テストが通ればよい" とゴールを定義し、AI に自由度を渡した。同じ発想は "既存製品のリライトを AI に任せる" 全ての実務で流用可能。(2) "thread-per-connection" で 30 年来の Postgres 設計をひっくり返す: Postgres は 1996 年設計の fork() ベースでコネクションごとにプロセスを立てるが、Rust の async ランタイム + Tokio 等で "1 プロセス多スレッド" に変えると、コネクション増加時のメモリ効率と connection pooling 常設化のコストが劇的に下がる。これは PgBouncer が長年埋めてきたギャップを言語処理系側で解消する話。(3) コストが個人レベルに落ちた: $1,600/月 × 数ヶ月で "SQL エンジンを丸ごと書き換え" が届く価格に。既存 OSS の "戦略的リライト" が新しい競争軸として現実的になった。

TDD, AI agents and coding with Kent Beck — Pragmatic Engineer podcast 全文 (無料公開)

📐設計論Kent BeckPragmatic Engineer
出典: The Pragmatic Engineer / Kent Beck 出演回

XP / Agile Manifesto / TDD の生みの親 Kent Beck が、AI エージェント時代における TDD の位置づけを語る。核となる主張は 3 つ: (1) TDD は AI エージェントに対する "superpower" — エージェントは容易にリグレッションを入れるが、単体テストが最小の防御線になる。(2) AI エージェントの mental model は "予測不可能な genie": 願いを叶えるが、しばしば予期せぬ (かつ非論理的な) 形で返してくる。(3) 我々は "コードは急速に累積しているのに、その信頼 (trust) の累積が追いついていない" — trust の累積速度を上げることが今後の SWE の主戦場だと Beck は捉える。関連して Beck は 2026 年 3 月に "Still Burning" というポッドキャストを開始。

💡Beck の "trust の累積が遅い" 指摘は、7/10 号 Fowler の "同じ変更を少ないトークンで済ませられる = 良いアーキテクチャ" と相補的。両者とも "AI が書いたコードを人間/組織が信頼する仕組み" を設計論の中核に据えている。汎用化可能な観察: 昨日の Bun / 本日の pgrust の "AI 書き換え" は、いずれも "既存の回帰テスト" を trust の基盤にした。つまり "テストが十分にある領域は AI に任せられる範囲が広く、テストがない領域は依然として人間中心" という現代的な境界線。読み手の実務では、"AI に任せたい領域" ほど先に "テストで trust を積む" ことが実装ロードマップの逆順で効いてくる — レガシー Rails/Django アプリの AI 移植を考えるなら、まず golden test の充実が最短距離。

🧠 Anthropic Platform — モデル世代交代のスケジュール

Anthropic API 移行スケジュール確定: Opus 4.7 fast mode 7/24 廃止、Opus 4.1 は 8/5 完全リタイア

🆕新リリースAnthropic⚠移行
出典: Claude Platform Docs (API リリースノート) / Anthropic newsroom

Anthropic API のリリースノートで、モデル世代交代の期日が具体化。(1) Opus 4.7 fast mode: 7/24 で除去、以降 speed:"fast" でのリクエストはエラー、移行先は Opus 4.8 fast。(2) Opus 4.1 (claude-opus-4-1-20250805): 8/5 で完全リタイア、移行先も Opus 4.8。(3) Cache diagnostics が public beta 化: diagnostics.previous_message_id をリクエストに載せると、prompt cache の hit/miss と "どこで prefix が乖離したか" の理由が返る、beta header は cache-diagnosis-2026-04-07。(4) agent-memory-2026-07-22 beta: メモリ一覧の順序が "server 側で定義した安定順" に、depth パラメータの選択肢が簡略化。(5) Claude Managed Agents で self-hosted sandbox が本番利用可能に。

PostgreSQL 19 Beta 1 — I/O ワーカ自動スケールと Autovacuum の並列化

🆕新リリースPostgreSQL📐設計論
出典: PostgreSQL Global Development Group (2026-06-04 リリース) / Publickey (2026-07-09 日本語解説)

PostgreSQL 19 Beta 1 が 6/4 に公開、日本語解説が 7/9 の Publickey に。中核変更: (1) PostgreSQL 18 で導入された非同期 I/O サブシステムを 19 が拡張、io_method=worker が io_min_workers / io_max_workers / io_worker_idle_timeout / io_worker_launch_interval に応じて I/O ワーカ数を自動スケール。(2) Autovacuum が並列ワーカで実行可能に (新パラメータ autovacuum_max_parallel_workers、テーブル単位で autovacuum_parallel_workers)、テーブル間の優先度付けの新スコアリングも導入。(3) 正式リリースは 2026 秋 (9〜10 月) 予定、Beta 2 / RC は継続予定。

💡設計論として重要なのは "Postgres の性能ボトルネックが徐々に "SQL エンジン" から "I/O と保守処理" に移った" のを 19 でも継続的に潰しに来ている点。汎用化可能な観察 — (1) Autovacuum の並列化は "肥大化した SaaS で夜間 vacuum が間に合わず断片化する" 典型パターンに直接効く。今まで DBRE が手動で VACUUM PARALLEL を叩いていたのを autovacuum に組み込んだ形。(2) I/O ワーカ自動スケールは Kubernetes 的 "HPA (Horizontal Pod Autoscaler)" 発想を DB プロセス内に持ち込んだ抽象。同様の "worker pool の自動サイジング" は Redis の I/O threads や Node.js の libuv でも見られる潮流で、"DB の水平垂直スケール判断を DB エンジン自身に委ねる" 方向。(3) 上の pgrust "thread-per-connection" と対比すると、Postgres コア陣営はプロセスモデルを維持しつつ I/O 層で追いつく戦略を明確化した — Rust リライトが未成熟なうちは公式 Postgres が現実解であり続ける。

Multigres v0.1 alpha — Vitess の作者が Supabase で作る "Postgres 版 Vitess"

🆕新リリースPostgreSQLVitess📐設計論
出典: Supabase Blog (2026-06-04 alpha 公開) / Publickey (2026-07-09 日本語解説)

YouTube を支えた Vitess (MySQL の水平スケール層) の共同発案者 Sugu Sougoumarane が Supabase に参画して立ち上げた OSS。v0.1 alpha (6/4 公開、7/9 に Publickey が日本語紹介) は "Postgres のシャーディング + コネクションプーリング + 自動フェイルオーバー + バックアップオーケストレーション" を一体で提供する "Postgres の OS" を目指す。アーキテクチャは MultiGateway (クエリルーティング) + MultiPooler (接続プール) のレイヤ型プロキシ、Kubernetes オペレータで運用、Apache 2 ライセンス。関連データの co-location を意識した relational sharding モデルも特徴。

🍎 プラットフォーム / 言語 — Apple が Swift をカーネル層に

Apple Internals: Swift in the Kernel — WWDC 2026 発表の "メモリセーフカーネル" 準備が macOS 26 に着弾

📐設計論AppleSwiftKernel
出典: Josh Maine (calif.io) 解析記事 / OSnews / Publickey (2026-07-07 日本語解説)

Apple が WWDC 2026 で発表した "macOS 27 系でカーネルの一部を Swift で書く" の第一段が macOS 26 SDK / ビルドで判明。(1) KernelKit という内部 SDK が新設、(2) 各 OS 用に別々の Mach-O プラットフォーム識別子が追加、(3) Embedded Swift ランタイムが macOS のカーネル拡張 (kext) に静的リンクされる形で同梱。ただし XNU コア (Mach / BSD / IOKit) は C/C++ のまま、Swift は "KernelKit の一部 kext 内の Embedded ランタイム" として限定的に出現するに留まる。iOS ではツールチェイン / ランタイム / リンク環境の下地整備が終わった段階で、Swift 実装のカーネル部品はまだ実装されていない。

💡設計論として重要なのは "既存の C カーネルを Swift で全書き換えする" のではなく、"kext (拡張ドライバ層) に Embedded Swift ランタイムを差し込む" 段階的移行を選んだ点。汎用化可能な観察 — (1) "Embedded Swift ランタイム" とは、Swift の言語機能から GC / ARC / 大きなランタイムライブラリを削ぎ落として "OS カーネルや MCU で使える最小構成" にしたモード。C/C++ を全置換せず、"新規実装だけ Swift、既存は C" と混在させられる。(2) 昨日の TypeScript 7.0 が採った "アルゴリズムは変えずランタイムだけ差し替える" 戦略と対照的で、Apple は "既存を保存しつつ、新規部分にだけメモリ安全な言語を導入する" 逆方向のリスク管理。両者とも "大規模書き換えを一気にやらない" 教訓を示す — Bun/pgrust の AI 一気書き換えとは違う流派。(3) 読み手の実務での応用: 自社レガシー (PHP / Perl / 古い Rails 等) を "全書き換え" するか "新機能だけ新言語" にするかの意思決定に、Apple 型は "コアは触らず境界層で切り替える" 参照モデルとして使える。特に "書き換え中に本番を止めない" 制約下では強い。

🔬 論文 / 研究

Reinforcement Learning for LLM-based Multi-Agent Systems through Orchestration Traces (arXiv 2605.02801)

🔬論文Multi-Agent📐設計論
出典: arXiv 2605.02801 (2026 年 5 月投稿、7 月に注目度が上がり多エージェント研究のリファレンスに)

"Orchestration traces" — マルチエージェント実行時に自然に生成される "サブエージェント spawn / 委任 / 通信 / tool use / return / 集約 / 停止判定" の時間順イベントグラフ — を RL の学習信号として使う手法。従来 "1 個の LLM を PPO 等で fine-tune する" 発想から、"エージェント群のオーケストレーション全体を報酬信号でチューニングする" 発想へ拡張。実行トレースを graph として保存し、成功/失敗経路の差分を報酬に写像する仕組みが中心。

💡汎用化可能な観察 — (1) "orchestration trace" は今後の LLM 運用の "アプリログ" 相当 になる。従来 Web アプリで APM / distributed tracing が業界標準になったのと同型で、"どのエージェントがどのツールを呼び、どの分岐でエラーが出たか" をトレースとして持たなければ、AI 運用の SRE ができない。(2) 昨日 7/10 号の Mnemosyne (Agentic Transaction Processing、"提案が commit 前に検証される") と、この論文の "trace ベース RL" は補完関係にある — Mnemosyne が "実行前ガード"、こちらが "実行後学習"。両方揃うと "AI に業務を任せる" 実務系のロードマップが (a) trust boundary を明示 → (b) 実行トレースを保存 → (c) RL で継続改善 の 3 段になる。(3) 実務適用の入口は Anthropic の Cowork / Skills / MCP の "実行ログ" を既にサーバ側に持っている点で、当面は "そのログから orchestration graph を組み立てる" 内製ツール実装が読み手のイベント運用領域でも初動になる。

国内の AI エージェント関連の "本日の動向" をまとめる連載の 7/11 号。主要テーマは "PoC から本番運用への移行を支える基盤 / 専門領域特化 / 業務データとの安全な接続"。具体的動き: 合同会社 Nagon が "AI エージェント向けデータ流通基盤 Nonagon Link" ベータ版を 7/10 から国内限定公開、日本情報クリエイトが "賃貸革命 11" のデータを Claude Desktop 等から自然言語で照会できる MCP サーバを 7/22 から試験提供、JTP が OpenAI 最新モデル (GPT-5.6 Sol / Terra / Luna) 対応を 7/10 完了。

💡読み手のイベント運用 × 社内業務 AI 活用の観点で最も直接的な日本語記事。汎用化可能な観察 — (1) "業務データ ↔ AI" の接続レイヤとして MCP がデファクトになっている — 賃貸管理という国内 SaaS 領域 (SIer 産業の中核) が Claude Desktop から MCP 経由で自然言語照会できるようになる、というのは "MCP は開発者だけの技術ではなく、業務系 SaaS の外側で採用が始まっている" 証拠。(2) データ流通基盤 (Nonagon Link) の登場: 従来「エージェントが業務データを触るには API を書け」の世界だったのが、"データ提供者・エージェント運用者・アプリ開発者を分離した SaaS" が生まれ始めている。海外の LlamaIndex / LangChain のホスト版と対比する読み手には、日本版のポジション取りとして興味深い。(3) MCP 対応 = 業務プロダクトの "第 3 の SDK" (Web SDK / モバイル SDK / MCP) という位置づけに整理できると、自社サービスの拡張ロードマップに組み込みやすくなる。

Claude Code と MCP で「記事を書く→Zenn 公開→経歴に蓄積」を自動化した (Zenn)

ZennClaude CodeMCP
出典: Zenn (muramasa0228)

Claude Code の Skills + MCP サーバを組み合わせて "記事執筆 → Zenn 公開 → 経歴データベースへ蓄積" を一気通貫で自動化した実装記事。核心は "人間の承認 (Push 前レビュー) を挟むが、それ以外は完全に AI が回す" 半自動フローで、"どの操作は人間承認、どの操作は AI 自律" を明確化した設計。関連して "Zenn 記事のズボラ運用術" 派生記事もあり、コミュニティ内で類似の Skill+MCP 実装が広がっている。

💡読み手のイベント運用視点への転用余地が大きい記事。汎用化可能なエッセンス — (1) "AI 自律で回すが、外向き publish の直前だけ人間承認" パターン は、7/10 号の Zenn team_zenn 記事の "Manual mode を publish 直前だけに絞る" 二層設計と同じ潮流。イベント通知や顧客告知の生成でも "文章生成は AI、送信は人間の承認" が最初の 1 週間で立てる典型構造。(2) "経歴データベースに蓄積"という発想が面白く、これは自社の "顧客対応履歴" や "イベント運営履歴" にも横展開できる。ワークフローの副産物としてナレッジベースが自動更新される設計は、7/9 号 Anthropic Reflect (memory 可視化) の背後にある思想と同じで、"エージェント経由の作業ログ = 組織の memory" として意識的に扱う設計は 2026 年後半のトレンド。(3) MCP の勉強コストが下がったので、今週動くなら "自社 CMS / Notion / Slack を叩く MCP サーバを 1 つ作る → Claude Code で書かせる" が最短経路。

Cavern Manticore — Iran-linked APT が SysAid の更新経路を悪用、モジュラー .NET C2 で IT/官公庁を標的

⚠セキュリティサプライチェーンAPT
出典: Check Point Research / The Hacker News / SecurityWeek

Iran の MOIS (情報安全保障省) 系統の APT クラスタが、新しいモジュラー C2 フレームワーク "Cavern (Cav3rn)" を使ってイスラエルの IT / 官公庁を狙う。攻撃連鎖は "SysAid の更新機能を悪用してトロイの木馬化した DLL (Cavern Agent) を実行 → C2 と通信して用途別モジュール (偵察 / データ窃取 / トンネリング / 横方向移動) を追加ダウンロード"。実装は .NET Framework / .NET Mixed-Mode C++/CLI / .NET Native AOT を混在させた成熟した構造。MuddyWater / Lyceum (OilRig 傘下) とも戦術重複あり。特徴は "IT ベンダを踏み台にして 2 段跳びで最終標的に到達する" サプライチェーン型侵入。

💡汎用化のエッセンス — (1) "IT サービスベンダの信頼関係を武器化する" 手口は、Cavern 固有ではなく 2020 年 SolarWinds、2022 年 Kaseya、2024 年 XZ backdoor に続く系譜。自社が "ベンダから何かを更新される側 (SysAid のような監視/バックアップ/RMM ツールを入れている)" なら、"更新経路も高権限資産として扱う" 意識設計が要る。(2) C2 が Native AOT を使い分けるのはフォレンジック回避目的で、"1 種類の実行体でしか動かない前提の EDR" は捕捉率が落ちる。テックリードの防御側視点では、EDR ルールを "特定バイナリタイプ限定" にせず、行動ベース (プロセスツリー、C2 通信パターン) に寄せる方向に更新する必要がある。(3) 昨日 7/10 号の JetBrains CVE 群と同じで、"開発 / 運用ツールが最上位の攻撃対象" というトレンドは加速。特に SysAid のような IT 運用系 SaaS は "セキュリティ製品と同じ扱いで監視" するのが 2026 の実務ライン。
