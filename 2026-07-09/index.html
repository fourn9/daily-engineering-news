出典: Windows Experience Blog / BleepingComputer / MacObserver

GitLost — 公開 Issue の本文だけで GitHub Agentic Workflows に private repo を漏洩させる prompt injection

⚠セキュリティPrompt InjectionGitHub
出典: Noma Security / The Register / The Hacker News / Dark Reading

Noma Labs が 7/6 開示。GitHub Agentic Workflows は同一 organization の public repo に立てられた Issue 本文を "信頼データ" として読み取る。攻撃者は Issue に自然文で命令を書き込むだけで、エージェントに private repo の内容を漏洩させられる。ガードレールは "Additionally" と一語前置きするだけで貫通した。認証・コード知識不要、Issue を書いて待つだけ。

💡Prompt Injection とは「LLM に外部データを読ませた瞬間、そのデータ内の指示文が上位命令として実行される」古典的脆弱性。設計論としての教訓は "信頼境界 (trust boundary)" の甘さで、エージェントに与える入力に対して "system 命令" と "untrusted user data" のラベルを最後まで保持しない実装が原因。汎用化可能なエッセンス — (1) 自社エージェントの入力ソースを "誰でも書き込める領域" (Issue/コメント/メール/DB のユーザ入力欄) と "内部制御領域" で二分し、前者由来のテキストにはツール実行権限を絞る。(2) "この文字列は行動指示ではなくデータである" を宣言する構造 (XML タグ / 別チャネル配信) が有効。GitHub の "Additionally" 事件は、ガードレールを "文言マッチ" ではなくアーキテクチャで担保するしかないことを示す実例。

Zero-Growth Stack, Real Gains: How Stack Allocation Can Save 10% CPU in Go (Uber)

📐設計論GoUber
出典: Uber Engineering Blog

Go の goroutine スタックはデフォルト 2KB で、溢れると倍に拡張 → データコピーというコストが発生する。Uber の高スループット系サービスではこの拡張サイクルだけで最大 10% の CPU を食っていた。CPU プロファイルとバイナリシンボルから "サービスごとの最適初期スタックサイズ" を自動計算する profile-guided automation を導入し、200 万コアの fleet 全体でメモリオーバヘッド 2% 未満、拡張オーバヘッド 1% 未満、CPU 効率 +16% を達成。

💡"デフォルト 2KB は Go 開発者が全 goroutine に払う "予防的最小コスト" だが、大量並列で長寿命な goroutine を大量に持つサービスでは "初回に一発大きく取っておく" ほうが割安" というのが要点。汎用化可能なエッセンスは "ランタイムの安全デフォルト" と "自社ワークロードの実測分布" のギャップを埋める profile-guided な設定生成 — 同型パターンとしては Go GC tuner (2022) や JVM の JIT ヒント、Rust の PGO 最適化がある。自社が同じ規模でなくても、"デフォルト値が支配的コストになるレイヤ" (例: DB コネクションプール初期サイズ、Kafka producer batch size) を CPU プロファイル駆動で決める設計は転用可能。

Hybrid Core Allocation: From Overallocation to Reliable Sharing (Uber Odin)

📐設計論SchedulerUber
出典: Uber Engineering Blog

Uber の stateful プラットフォーム Odin (380 万コンテナを扱う) の縦スケーラは、バースト性ワークロードで "専有コア割当" 一辺倒だと過剰プロビジョンになりがちだった。ベースライン性能用の dedicated core と、バースト用の shared core を組み合わせるハイブリッドモデルを Linux cpusets / cpu.shares で実装。可変ワークロードのオーバプロビジョン削減とフリート全体の CPU 利用率向上を、SLO 保証・クロスホスト移送なしで達成。

💡"burstable" 概念自体は K8s の QoS class (Guaranteed/Burstable/BestEffort) や AWS EC2 T シリーズで既知だが、これを "自社スケジューラの中で dedicated と shared を 1 pod で両立する" 設計にした点が独自。汎用化可能なエッセンスは "SLO 保証は dedicated で担保、余剰は shared で吸収" という 2 段構成 — 自社で K8s を運用しているなら、"CPU limit + request を厳密に張る + cpu-manager-policy=static + HPA" の組み合わせで似た挙動を作れる。Odin 特有なのは "cross-host workload migration なし" の要件で、これはステートフル (DB/キャッシュ) を扱う前提から来る制約。ステートフル系を大規模に運用する組織で有効な参照実装。

☁️ データ基盤 / DB

PostgreSQL 19 Beta ディープダイブ — SQL/PGQ グラフクエリ・並列 autovacuum・REPACK/ON CONFLICT DO SELECT

🆕新リリースPostgreSQL📐設計論
出典: Snowflake Engineering Blog / InfoQ / PostgreSQL.org

Postgres 19 (Beta 1 は 6/4、GA は 9〜10 月見込) の設計ポイントを Snowflake が解説。既存リレーショナルテーブル上に SQL/PGQ (プロパティグラフクエリ) を追加、Neo4j 等の別 DB を持たずグラフクエリが書ける。ON CONFLICT DO SELECT で "insert して結果を返す or 既存を返す" の get-or-create セマンティクスを 1 statement で実現。REPACK でオンライン領域再構築、autovacuum に parallel worker、pg_plan_advice/pg_stash_advice でプラン制御。JIT はデフォルト OFF、TOAST 圧縮は lz4 が既定に。

💡SQL/PGQ (SQL:2023) は "リレーショナルスキーマ上に PROPERTY GRAPH を "ビュー" として定義してグラフパターン (MATCH (:User)-[:FRIEND]->(:User)) で問い合わせる" 標準規格。既存 DB を捨てずグラフクエリを載せられるのが要点で、"軽量に社内グラフ分析を始めたいが Neo4j 導入は重い" 中規模事例の受け皿になる。汎用化の観点では ON CONFLICT DO SELECT が地味に効く — 従来 "INSERT ... ON CONFLICT DO NOTHING → SELECT" と 2 クエリ + 競合レースを書いていた get-or-create が atomic 1 文で書ける。JIT デフォルト OFF は "ワークロードによっては JIT のオーバヘッドが利得を食う" 実運用の学びを反映した正常化。

🔬論文Multi-Agent

PostgreSQL 19 beta 版リリース — Autovacuum の並列化と I/O worker の自動スケーリング (Publickey)

🆕新リリースPostgreSQLPublickey
出典: Publickey (2026-07-09)

上の英語版と同じ Postgres 19 Beta を日本語で概説。autovacuum が並列 worker 対応で大テーブルの vacuum が時間短縮、io_method=worker が io_min_workers / io_max_workers で自動スケール。オンラインで data checksum の有効化・無効化が可能に (クラスタ再初期化不要)。

💡日本の運用現場で効くのは "autovacuum の並列化" と "オンライン checksum" の 2 つ。前者は "巨大テーブルの autovacuum が業務時間に食い込む" 従来の悩みへの直接的な解、後者は "運用中にデータ破損検知を有効化したいがクラスタ停止できない" という古典的ジレンマの解消。汎用化観点では "オンライン化 (無停止での設定変更)" が Postgres 19 全体のテーマで、Aurora / Cloud SQL 等マネージド DB の "無停止運用" 機能を OSS 本体が追随しているという構造的意味合い。
